# Design Specification: Diesel SQLite Migration + MVVM Architecture

**Date:** 2026-06-12  
**Author:** Claude Sonnet 4.5  
**Status:** Approved  

## Executive Summary

Replace DataFusion/Parquet storage with Diesel SQLite ORM and implement MVVM architecture with conditional background threading. Fresh database start (no migration from Parquet). Asupersync for async I/O tasks, standard channels for UI communication. Comprehensive unit tests for business logic.

## Objectives

1. **Remove DataFusion dependency** — Replace `datafusion_bingimage.rs` with Diesel SQLite
2. **Implement MVVM pattern** — Separate UI from business logic with ViewModel
3. **Conditional threading** — Background thread for GUI/Android, synchronous for CLI
4. **Async with Asupersync** — Structured concurrency for downloads and I/O
5. **Comprehensive testing** — Unit tests for all business logic, integration tests for entry points

## Background

### Current State

- **Database**: DataFusion with Parquet files (~1300 lines in `datafusion_bingimage.rs`)
- **Threading**: Mixed — some async with tokio, UI tightly coupled with business logic
- **Entry points**: Desktop GUI, Android, CLI, WASM
- **Testing**: Basic tests in DataFusion module

### Problems

- DataFusion is heavyweight for simple CRUD operations
- No separation between UI and business logic

### Connection Management (mod.rs)

**Why**: Single-threaded SQLite with WAL mode and busy_timeout (following uad-shizuku pattern). No connection pool needed for current use case.

```rust
use diesel::prelude::*;
use diesel_migrations::{embed_migrations, EmbeddedMigrations, MigrationHarness};

const MIGRATIONS: EmbeddedMigrations = embed_migrations!();

pub fn establish_connection(db_path: &Path) -> SqliteConnection {
    let url = db_path.to_str().expect("Valid UTF-8 path");
    let mut conn = SqliteConnection::establish(url)
        .unwrap_or_else(|_| panic!("Error connecting to {}", url));

    #[cfg(not(target_arch = "wasm32"))]
    {
        // Enable WAL mode for better concurrent access
        diesel::sql_query("PRAGMA journal_mode=WAL;")
            .execute(&mut conn)
            .expect("Failed to set WAL mode");

        // Set busy timeout to 30 seconds
        diesel::sql_query("PRAGMA busy_timeout=30000;")
            .execute(&mut conn)
            .expect("Failed to set busy timeout");

        // Run migrations
        conn.run_pending_migrations(MIGRATIONS)
            .expect("Failed to run database migrations");
    }

    conn
}
```

### WASM Compatibility

**Why**: WASM builds need stub implementations since SQLite isn't available in browser.

**How to apply**: Use `#[cfg(not(target_arch = "wasm32"))]` for real implementation, provide stub returns for WASM builds.

### ViewModel Structure

**Why**: Conditional compilation separates async (GUI/Android) from sync (CLI) modes. Single codebase, different runtime behavior.

```rust
use std::sync::mpsc::{Sender, Receiver, channel};
use std::path::PathBuf;
use anyhow::Result;

pub struct ViewModel {
    db_path: PathBuf,
    
    // GUI/Android: channels for async communication
    #[cfg(not(feature = "cli-only"))]
    command_tx: Option<Sender<ViewModelCommand>>,
    
    #[cfg(not(feature = "cli-only"))]
    event_rx: Option<Receiver<ViewModelEvent>>,
}

impl ViewModel {
    // GUI/Android: spawns background thread with Asupersync runtime
    #[cfg(not(feature = "cli-only"))]
    pub fn new_async(db_path: PathBuf) -> Result<Self> {
        let (cmd_tx, cmd_rx) = channel();
        let (evt_tx, evt_rx) = channel();
        
        let db_path_clone = db_path.clone();
        std::thread::spawn(move || {
            background::run_background_loop(db_path_clone, cmd_rx, evt_tx);
        });
        
        Ok(Self {
            db_path,
            command_tx: Some(cmd_tx),
            event_rx: Some(evt_rx),
        })
    }
    
    // CLI: direct synchronous execution, no thread
    #[cfg(feature = "cli-only")]
    pub fn new_sync(db_path: PathBuf) -> Result<Self> {
        Ok(Self { db_path })
    }
    
    // GUI/Android: send command to background thread
    #[cfg(not(feature = "cli-only"))]
    pub fn send_command(&self, cmd: ViewModelCommand) -> Result<()> {
        self.command_tx.as_ref()
            .expect("command_tx initialized")
            .send(cmd)?;
        Ok(())
    }
    
    // GUI/Android: poll for events from background thread
    #[cfg(not(feature = "cli-only"))]
    pub fn poll_events(&self) -> Vec<ViewModelEvent> {
        self.event_rx.as_ref()
            .expect("event_rx initialized")
            .try_iter()
            .collect()
    }
    
    // CLI: synchronous operations (direct calls to commands.rs)
    #[cfg(feature = "cli-only")]
    pub fn download_images_sync(&self, market_code: &str) -> Result<usize> {
        use crate::db;
        let mut conn = db::establish_connection(&self.db_path);
        commands::download_images_sync(&mut conn, market_code)
    }
    
    #[cfg(feature = "cli-only")]
    pub fn get_images_by_status_sync(&self, status: ImageStatus) -> Result<Vec<BingImage>> {
        use crate::db;
        let mut conn = db::establish_connection(&self.db_path);
        db::operations::get_images_by_status(&mut conn, status)
    }
    
    // Additional CLI sync methods...
}
```

### Background Thread (viewmodel/background.rs)

**Why**: Isolates Asupersync runtime to background thread. UI thread never blocks on I/O.

**How to apply**: GUI/Android call `send_command()` and `poll_events()` in their render loops. Background thread processes commands using Asupersync for I/O.

```rust
use std::sync::mpsc::{Receiver, Sender};
use std::path::PathBuf;
use asupersync::runtime::RuntimeBuilder;
use asupersync::Cx;

pub fn run_background_loop(
    db_path: PathBuf,
    cmd_rx: Receiver<ViewModelCommand>,
    evt_tx: Sender<ViewModelEvent>,
) {
    // Create Asupersync runtime for I/O tasks
    let runtime = RuntimeBuilder::current_thread()
        .build()
        .expect("Failed to create Asupersync runtime");
    
    let mut conn = crate::db::establish_connection(&db_path);
    
    // Message loop
    for cmd in cmd_rx {
        match cmd {
            ViewModelCommand::DownloadImages { market_code } => {
                // Spawn async task in Asupersync runtime
                let result = runtime.block_on(
                    runtime.handle().spawn(async move {
                        let cx = Cx::current().expect("Asupersync context");
                        download_images_task(&cx, &market_code).await
                    })
                );
                
                match result {
                    Ok(images) => {
                        // Save to database
                        for img in &images {
                            if let Err(e) = crate::db::operations::upsert_image(&mut conn, img) {
                                evt_tx.send(ViewModelEvent::Error { 
                                    message: format!("DB error: {}", e) 
                                }).ok();
                            }
                        }
                        evt_tx.send(ViewModelEvent::DownloadComplete { 
                            count: images.len() 
                        }).ok();
                    }
                    Err(e) => {
                        evt_tx.send(ViewModelEvent::Error { 
                            message: format!("Download failed: {}", e) 
                        }).ok();
                    }
                }
            }
            
            ViewModelCommand::GetImagesByStatus { status } => {
                match crate::db::operations::get_images_by_status(&mut conn, status) {
                    Ok(images) => {
                        evt_tx.send(ViewModelEvent::ImagesLoaded { images }).ok();
                    }
                    Err(e) => {
                        evt_tx.send(ViewModelEvent::Error { 
                            message: format!("Query failed: {}", e) 
                        }).ok();
                    }
                }
            }
            
            ViewModelCommand::Shutdown => break,
            
            // Other command handlers...
        }
    }
}

// Asupersync task for downloading images with structured concurrency
async fn download_images_task(
    cx: &Cx,
    market_code: &str,
) -> Result<Vec<crate::db::models::NewBingImage>, anyhow::Error> {
    use asupersync::Outcome;
    
    // Spawn parallel download tasks (structured concurrency)
    let mut tasks = vec![];
    for i in 0..8 {
        let mc = market_code.to_string();
        let task = cx.spawn(move |task_cx| async move {
            download_single_image(&task_cx, &mc, i).await
        })?;
        tasks.push(task);
    }
    
    // Join all tasks (guaranteed cleanup on cancel/error)
    let mut results = vec![];
    for task in tasks {
        match task.join(cx).await {
            Ok(Outcome::Ok(data)) => results.push(data),
            Ok(Outcome::Cancelled) => {
                log::warn!("Download task cancelled");
            }
            Err(e) => {
                log::error!("Download task failed: {}", e);
            }
        }
    }
    
    Ok(results)
}
```

## Error Handling

**Strategy**: Use `anyhow::Result<T>` throughout for error propagation. Convert to ViewModelEvent::Error at boundaries.

**Why**: Matches existing codebase patterns. Flexible error types for business logic, user-friendly messages at UI boundary.

**How to apply**:
- Database operations return `anyhow::Result<T>`
- Background thread catches errors, sends `ViewModelEvent::Error`
- UI displays error messages to user
- CLI prints errors to stderr and exits with non-zero code

## Non-Goals

- ❌ Data migration from Parquet to SQLite (fresh start)
- ❌ Connection pooling (single-threaded SQLite sufficient)
- ❌ Unit tests for UI dialogs (require UI framework)
- ❌ Unit tests for platform bindings (require devices)
- ❌ Performance optimization beyond indexes (premature)

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Asupersync instability (experimental crate) | High | Extensive testing, fallback plan to use tokio if needed |
| SQLite locking on Android | Medium | WAL mode + busy_timeout, test on real devices early |
| Breaking changes to existing UI | High | Preserve API surface, update call sites incrementally |
| Test coverage gaps | Medium | Code review checklist, run coverage reports |
| CI failures on some platforms | High | Test locally on each platform before PR |
